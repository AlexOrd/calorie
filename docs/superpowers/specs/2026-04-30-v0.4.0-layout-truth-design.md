# v0.4.0 — Layout & Truth Pass

**Date:** 2026-04-30
**Version bump:** 0.3.0 → 0.4.0 (minor)

## Goal

Tidy up surfaces that have drifted (Dashboard chrome, Stats hero, 90-day grid, Profile rows), drop image-export weight, and make the help modal honest by promoting body-fat into the BMR pipeline so every metric shown in the app actually drives something.

Each item is a separate commit; the release ships as one PR with a single changelog entry under `0.4.0`.

### 4 — Stats hero — no orphan tile

**Why:** When BMI + Body-Fat + Water all render, the mobile 2-col grid leaves Water alone in row 2 with an empty slot beside it.

**Changes:**

- `src/components/EnergyBalanceRow.svelte` (variant `'full'`, the BMI + Fat + Water sub-stats grid):
  - Keep parent `grid grid-cols-2 gap-2 sm:grid-cols-3`.
  - Add to **each child tile's** class list: `last:[&:nth-child(odd)]:col-span-2 sm:last:[&:nth-child(odd)]:col-span-1`.
  - Effect: when the last rendered tile lands in an odd position on mobile (i.e., alone in its row), it spans both columns; on `sm+` (3-col) the override is neutralized.

**Cases verified:**

| Tiles rendered    | Mobile (2-col)                              | sm+ (3-col)                          |
| ----------------- | ------------------------------------------- | ------------------------------------ |
| BMI + Fat + Water | row 1: BMI, Fat / row 2: Water (col-span-2) | one row, three cols                  |
| BMI + Water       | one row, both fit                           | one row + empty 3rd col (acceptable) |
| Fat + Water       | same                                        | same                                 |
| Water only        | full width                                  | acceptable                           |

If `sm+` empty-3rd-col is also a concern, future cleanup; for this pass, the mobile orphan is the visible bug.

### 5 — Heatmap — calendar fill, first-of-month highlight

**Why:** Current grid is column-major (`grid-flow-col grid-rows-7`) and starts with leading-padded weekdays. User wants a row-major calendar starting at top-left = oldest day.

**Changes in `src/components/Heatmap.svelte` 90-day section:**

- Replace `gridCells` derivation:
  - Build the date list `[today-89, …, today]` (oldest first; matches user's "filled in right" direction).
  - **No leading pad** — oldest day is at top-left of the grid regardless of its weekday. (Weekday alignment is intentionally dropped: the user described the grid as a sequential count of days, not a calendar with weekday columns.)
  - **Trailing pad** to round out the last row to a multiple of 7: `pad = (7 - (90 % 7)) % 7 = 1` `null` cell after today, so the rendered rectangle has no ragged right edge.
- Change grid container: `grid grid-cols-7 grid-flow-row gap-1` (drop `grid-flow-col grid-rows-7` and `overflow-x-auto`).
- Cell render: when `dateFromKey(cell.key).getDate() === 1`, add `ring-1 ring-fg/40 ring-inset` to the cell's class list. Title text becomes `"{cell.key} (1-е) — {label}"` so the highlight has tooltip context.
- Drop `overflow-x-auto pb-1` — the row-major grid no longer needs horizontal scroll.
- Cell sizing: keep `h-3 w-3` but the cells now sit in a 7-column grid. With 7 columns × 12px + 6 gaps the grid is ~110px wide; let the grid stretch via `aspect-square` cells and `grid grid-cols-7 gap-1` (no fixed cell size) so the grid fills the card width on all viewports. Replace `class="h-3 w-3 …"` with `class="aspect-square …"` on each cell — same shape, fluid width.

**Verification:**

- 90 days % 7 = 6, so 13 full rows (oldest 91 cells of which 1 is trailing-pad after today). Top-left cell = `today − 89`. Bottom-right cell of row 13 = trailing-pad placeholder; today is one cell to its left.
- Day-of-month=1 cells in the 90-day window (Feb 1, Mar 1, Apr 1 with today = Apr 30) each get the ring.

The 13-day Categories and Hydration strips are untouched — they're already row-major-equivalent (single row of 13).

### 6 — Body-fat → Katch–McArdle, StatsHelpModal as a legend

**Why (truth):** Body-fat is currently display-only — `bodyFatPct()` is invoked once for the Stats hero "Жир %" tile and feeds nothing. Mifflin–St Jeor doesn't read body composition. Promoting to **Katch–McArdle** when body-fat is computable makes the Жир tile drive BMR (and therefore TDEE, k_factor, daily targets, projection).

**Why (legend):** The help modal currently uses `text-accent` for every section header, regardless of where each metric appears in the UI. Replacing those headers with swatches that match each metric's actual on-screen colour turns the modal into a true legend.

**Calculation changes:**

- `src/lib/energy.ts`:
  - Add `bmrKatch(weightKg: number, bfPct: number): number` returning `370 + 21.6 * weightKg * (1 - bfPct / 100)`.
  - Add `bmrMifflin(p: ProfileInput): number` (rename of current logic).
  - Refactor `bmr(p)` to: try body-fat from profile via `bodyFatPct(...)` (using `waist_cm`, `neck_cm`, optional `hip_cm`); if numeric → return `bmrKatch(p.weight, bfPct)`; else `bmrMifflin(p)`.
  - Export a small helper `bmrFormula(p): 'mifflin' | 'katch'` for the help modal "current profile uses X" line.
- `src/lib/scaling.ts`:
  - Delete the local private `bmr()` and `tdee()` (currently duplicates of `energy.ts`).
  - Import `bmr` from `$lib/energy`. `tdee()` becomes `bmr(p) * p.activity` defined locally or inlined.
  - `BASELINE` (used for `computeKFactor`) is also a `ProfileInput` without body-fat fields, so it correctly uses Mifflin — no change in baseline behaviour.
  - `dailyTargets()` and `computeKFactor()` automatically pick up Katch when body-fat is present (the only change observed by callers).
- `src/components/EnergyBalanceRow.svelte`:
  - The Жир tile already exists; no change to the tile itself, but it's now meaningfully connected to the BMR cascade.

**Help modal rewrite — `src/components/StatsHelpModal.svelte`:**

Replace the current sections with the legend layout below. Each section's heading uses a coloured swatch (`<span class="h-3 w-3 rounded-full" style="background: var(--color-X);"></span>`) plus the section title in `text-fg` (drop the all-accent uppercase headers).

Sections (top → bottom):

1. **Калорії** (accent swatch) — what the kcal tile is, target = TDEE.
2. **Білок / Вуглеводи / Жири** (danger / warn / fat-blue swatches, three subsections) — `1.6 g/kg` (active ≥ 1.55) or `1.2 g/kg`; carbs = 50% kcal / 4; fat = 30% kcal / 9.
3. **BMR** — Mifflin formula + Katch–McArdle formula + "**Використовується Katch коли заповнені талія та шия (♀ також стегна).** Інакше — Mifflin." Inline shows what's active for the current profile (`{bmrFormula(profile.value)}`).
4. **TDEE** + activity multipliers (1.2 / 1.375 / 1.55 / 1.725).
5. **Спалено за день** (existing) — keep formula and breakdown text.
6. **Енергобаланс** — text + three swatches: accent (Дефіцит), muted (Баланс ±100 ккал), warn (Профіцит).
7. **Жир %** (warn swatch) — US Navy formula. Note: drives BMR via Katch when waist + neck (♀ also hip) are filled.
8. **BMI** — formula + four BMI-class swatches (muted / ok / warn / accent).
9. **Гідрація** — text + three swatches: danger (Нестача), ok (Норма), fat-blue (Понад норму). Note severe-deficit = < 50% target adds a category strike.
10. **Категорії А–З + k_factor** — what 100% means; `k_factor = clamp(TDEE/baseline_TDEE, 0.6, 1.6)`; scales `max_g` for gram-unit foods, leaves piece-unit foods alone.
11. **Календар (90 днів)** (existing) — three verdict swatches.
12. **Стрики** — kcal-deficit streak: day with delta < −100 kcal extends; delta ≥ −100 breaks. Water streak: day with `waterMl ≥ target` extends.
13. **Прогноз цілі** — 7-day rolling average daily Δ kcal → kg/day → ETA.

Swatch helper: define `<span class="inline-block h-3 w-3 rounded-full" style="background: var(--color-{name});"></span>` inline; no new component.
