# Calorie v0.2.0 release — combined design

**Date:** 2026-04-29
**Status:** spec
**Drivers:** `reserch.md` (research audit), user request for changelog system
**Supersedes:** [`2026-04-29-research-integration-design.md`](./2026-04-29-research-integration-design.md)

## Overview

This release bundles two parallel feature sets that ship as version `0.2.0`:

- **Feature set A — Research integration.** Add hydration tracking, BMI, Stats tips, and documentation citations driven by a research audit (`reserch.md`).
- **Feature set B — Changelog system.** Add a typed changelog source, automatic version derivation, two modals (auto-popup "What's new" and full Profile-accessed changelog), a Sparkles icon next to the Profile name, and a CLAUDE.md discipline section so future LLM-driven feature work updates the changelog.

The two sets are merged into one spec because they share a release, a CLAUDE.md edit, and a seeding dependency: the v0.2.0 changelog entry's content is the user-facing summary of feature set A.

## Feature set A — Research integration

### A.1 Context

A research dossier (`reserch.md`) audited the app's formulas against WHO / EFSA / CDC / Ukrainian MoH evidence. The core math (Mifflin-St Jeor BMR, PAL multipliers, Atwater 4-9-4, 1.2–1.6 g/kg protein, 50/30/~20 macro split) is already aligned with the literature. **No formula changes** to existing code are required.

The research surfaces four tractable gaps closed in this release:

1. **Hydration is unmeasured.** EFSA mandates 2.0 L ♀ / 2.5 L ♂; a weight-based 30 ml/kg refinement matches the same target band.
2. **BMI is uncomputed**, despite being the WHO foundational screening metric and despite all inputs already being collected.
3. **The Stats page exposes derived numbers without explaining them.** Users can't trace `1846 kcal` back to BMR × PAL.
4. **Documentation lags the research base.** Some research-validated formulas (IBW, safety floors, MoH 1073/1613 caps, pediatric percentiles) are out of scope for code today, but the citations are worth preserving as a future-proofing reference.

### A.2 Goals

- Add water tracking with a profile-derived target, surfaced on Activity, Dashboard, Stats, and the Heatmap.
- Add BMI as a small derived stat on the Stats page.
- Add inline, collapsible tips next to each derived metric on the Stats page.
- Update `docs/formulas.md` and `docs/food-database.md` with research citations and the two new implemented formulas (BMI, hydration target).
- Create `docs/health-references.md` to capture research-validated formulas the app does not yet implement.

### A.3 Non-goals

- IBW computation in code (Devine / Robinson / Miller / Hamwi) — documented only.
- Caloric safety-floor enforcement (1200 ♀ / 1500 ♂) — no deficit-target feature exists.
- Trans-fat / sodium / sugar tracking — would require new fields across the food DB; out of scope.
- Pediatric percentile mode — single-adult tracker.
- 5th PAL tier (1.9 "extremely active") — YAGNI for a personal app.
- Hydration in the energy-balance math — water has 0 kcal and stays orthogonal.

### A.4 Decisions

#### Hydration semantics

Hydration is a **separate axis** from energy balance, but it **does** participate in the daily heatmap verdict (severe under-hydration = 1 strike). It does not affect the energy-balance pill or `actualBurn()`.

#### Hydration target

```
target_ml = max(weight_kg × 30, gender_floor)
gender_floor = 2500 ml (male) | 2000 ml (female)
```

The 30 ml/kg coefficient is the conservative low end of the EFSA-aligned 30–35 ml/kg band; 35 is reserved for active climates and is intentionally not modeled. The gender floor protects very small users (a 50 kg woman would otherwise target 1500 ml — below the EFSA absolute).

#### Heatmap strip thresholds

| State    | Condition                  | Color              |
| -------- | -------------------------- | ------------------ |
| deficit  | `consumed < 70 % × target` | red / accent-warn  |
| balanced | `70 % ≤ consumed ≤ 120 %`  | green / accent-ok  |
| surplus  | `consumed > 120 %`         | blue / accent-info |

#### Day-verdict integration

`dayVerdict()` (currently local to `Heatmap.svelte`) gains one strike when hydration is severely deficient (`< 50 % × target`). The 50 % cutoff is intentionally tight — 60 % wouldn't punish a normal day, but a 40 % day is a genuine bad-health signal. The hydration strike adds to the same `overCount` already used for category strikes, so a clean food day with severe dehydration becomes amber instead of green.

Signature change (function stays local, not exported):

```ts
// before
function dayVerdict(entries: LogEntry[]): DayVerdict;
// after
function dayVerdict(entries: LogEntry[], hydrationDeficitSevere: boolean): DayVerdict;
```

#### Water input UX

Activity card: typed `number` input + `+250 ml` quick-add button. Mirrors the Steps card structure (typed input + percent + bar) while making the dominant interaction (hit the glass button) one tap.

#### BMI

Pure derived value, no state, no persistence. Rendered as a small pill at the start of `EnergyBalanceRow` on the Stats page (left of the existing balance pill, right of the new water pill — so the row reads `[BMI 23.4] [Δ −120] [💧 1.8 / 2.4 L]`). One collapsible tip explains the formula and the four WHO classes (underweight / healthy / overweight / obese; class subdivisions deferred). The BMI pill is variant-gated to `variant="full"` so it appears only on Stats, not on the Dashboard's compact row.

#### Stats tips

Native HTML `<details>` styled to look polished:

- Custom chevron via `summary::after` rotation.
- Smooth height animation via CSS `grid-template-rows: 0fr → 1fr` (no JS, no Melt).
- Subtle border + accent tint on open.
- `prefers-reduced-motion` collapses the animation.

Five tips, all in Ukrainian:

- **BMR** — _"Скільки калорій тіло спалює у спокої. Формула Mifflin-St Jeor."_ + formula.
- **TDEE** — _"BMR × множник активності — твоя добова норма для підтримки ваги."_ + formula.
- **Energy balance** — _"Спожито − спалено. Зелена зона ±100 ккал."_ + formula.
- **Hydration** — _"Ціль: 30 мл × вага, мінімум 2.0 / 2.5 л."_ + formula.
- **Day-verdict colors** — _"Зелений: усі категорії ≤ 100 %. Жовтий: 1–2 перебір. Червоний: 3+. Гідрація < 50 % додає страйк."_

#### Documentation strategy

`docs/formulas.md` owns implemented constants. New file `docs/health-references.md` owns research-validated formulas the app does not currently implement.

#### Data flow — hydration

```
Activity.svelte           ┐
EnergyBalanceRow.svelte   ├─→ activity store (waterMl) ─→ storage (activity_YYYY-MM-DD)
                          │                          ┌─→ hydrationTarget(profile)
Heatmap.svelte            ┴─→ profile store ─────────┘   hydrationState(...)
                                                         dayVerdict(entries, severeDeficit)
```

#### Storage migration

`activity.svelte.ts` `migrate()` is the single migration site. Add: if raw record has no `waterMl`, default to `0`. The existing `LegacyActivity` pattern already handles forward-compat. No new storage key. No schema-version bump needed.

### A.6 Edge cases

- **Profile saved before water target exists** — `hydrationTarget(profile)` is pure; runs against current profile.
- **No water entered for the day** — `waterMl = 0`, hydration state = `deficit`, `dayVerdict` adds a strike. Acceptable: matches user-stated semantics.
- **Day before water tracking existed** — historical `activity_YYYY-MM-DD` records load with `waterMl = 0` after migration; the heatmap will show a deficit strip and a strike. Acceptable; same retroactive-using-current-profile caveat already documented for energy balance applies.
- **Profile with no weight** — onboarding requires weight; gate hydration UI on `profile.value` being loaded.
- **Reduced-motion** — `<details>` height animation collapses to instant.
- **Telegram CloudStorage** — `waterMl` is one extra integer; well within the 4 KB / key budget.

## Feature set B — Changelog system

### B.1 Goals

- Maintain a typed list of versions and per-version items (`major` / `feature` / `fix`).
- Auto-derive the displayed app version from the topmost changelog entry — no separate manual version edit at runtime.
- Show a celebratory "What's new" modal on Dashboard mount when the user has seen a previous version but not the current one. Never show on first-ever launch.
- Show a full changelog modal on demand via a Sparkles icon on the Profile page.
- Persist the last-seen version through the existing storage abstraction (Telegram CloudStorage / LocalForage).
- Add a "Changelog discipline" section to CLAUDE.md so future LLM-driven feature work writes a changelog entry as part of the work.

### B.2 Non-goals

- Auto-bumping versions via Husky / Conventional-Commits parsing.
- Per-language changelog (Ukrainian-only).
- Search / filter inside the full changelog modal.
- "Coming soon" / preview entries.
- Roll-back behavior for downgrades (handled defensively as a no-op).

### B.3 Decisions

#### Source of truth

Single source of truth: **`src/data/changelog.ts`**. The displayed app version derives from `changelog[0].version`. `package.json` is kept in sync (Vite reads it via `define` for telemetry/build scripts) but the runtime does **not** depend on it — the runtime reads `changelog[0].version` directly. Adding a changelog entry IS the version bump; one edit per release, no drift possible.

#### Item types

```ts
export type ChangelogItemType = 'major' | 'feature' | 'fix';

export interface MajorItem {
  type: 'major';
  title: string; // Ukrainian
  body: string; // Ukrainian, 1–2 sentences
  icon?: keyof typeof LUCIDE_ICON_WHITELIST; // defaults to 'Sparkles'
}
export interface MinorItem {
  type: 'feature' | 'fix';
  text: string; // Ukrainian one-liner
}
export type ChangelogItem = MajorItem | MinorItem;

export interface ChangelogEntry {
  version: string; // 'MAJOR.MINOR.PATCH'
  date: string; // 'YYYY-MM-DD'
  items: readonly ChangelogItem[];
}
```

The icon whitelist (a small explicit map of allowed Lucide names) keeps tree-shaking honest.

#### v0.2.0 seed entry

```ts
{

  date: '2026-04-29',
```

#### Storage

One key, one string: `last_shown_changelog_version`. Reuses `storage` abstraction.

#### Trigger logic (Dashboard mount)

```
on Dashboard onMount (one-shot per page lifetime):
  if profile not loaded: return
  stored = await storage.load<string|null>('last_shown_changelog_version', null)
  if stored === null:
    storage.save('last_shown_changelog_version', APP_VERSION)   // silent first-launch write
    return
  if cmpVersion(stored, APP_VERSION) >= 0:
    return                                                       // already seen, or downgrade no-op
  entries = changelog.filter(e => cmpVersion(e.version, stored) > 0)
  popupItems = entries.flatMap(e => e.items).filter(i => i.type !== 'fix')
  if popupItems.length === 0:
    storage.save('last_shown_changelog_version', APP_VERSION)
    return
  openWhatsNewModal(popupItems)                                  // user sees major + feature items
  // on close:
  storage.save('last_shown_changelog_version', APP_VERSION)
```

`cmpVersion` is a pure function that splits each version on `.`, parses each segment to int, and returns -1/0/+1 from a 3-tuple compare.

#### State store

`src/state/changelog.svelte.ts`:

```ts
let _lastShown = $state<string | null>(null);
let _loaded = $state(false);

export const changelogState = {
  get lastShownVersion(): string | null { return _lastShown; },
  get isLoaded(): boolean { return _loaded; },
  get hasUnseenChanges(): boolean {
    if (!_loaded) return false;
    if (_lastShown === null) return false;       // first-launch is silent
    return cmpVersion(_lastShown, APP_VERSION) < 0;
  },
  async load(this: void): Promise<void> { … },
  async markSeen(this: void): Promise<void> { … },
};
```

The Sparkles icon next to the Profile name reads `hasUnseenChanges` and shows a small accent dot when `true`.

#### UI — WhatsNewModal

`src/components/WhatsNewModal.svelte` — centered card, backdrop blur, scale-in. Confetti burst on open (`burstConfetti(modalEl)`); success haptic. Layout:

- Backdrop: `bg-black/40 backdrop-blur-sm`, click-to-dismiss.
- Card: `bg-surface border-border` rounded-2xl, `max-w-md`, scale-in 200 ms, `prefers-reduced-motion` honored (instant).
- For each `major` item: hero card with Lucide icon (per-item `icon`, default `Sparkles`), big title, body.
- Below: "Також у цій версії:" + tight list of `feature` items.
- Close: `X` top-right + tap backdrop + Telegram BackButton when present (mounted/unmounted via the same handler-bracket pattern as Onboarding).

#### UI — ChangelogModal

`src/components/ChangelogModal.svelte` — same shell as `WhatsNewModal` but renders the full `changelog` array, all three item types, no confetti, no haptic celebration. Header per release: `v0.2.0 · 29 квіт. 2026`. Lazy-loaded via dynamic import to avoid bloating the main bundle.

#### Profile icon

`src/routes/Profile.svelte` — Sparkles icon button to the right of the user's name. Shows accent dot if `hasUnseenChanges`. Tap opens `ChangelogModal`. Haptic `selection` on tap. After `ChangelogModal` opens, call `changelogState.markSeen()` so the dot clears regardless of whether the auto-popup ever fired (e.g., user opened changelog manually first).

#### CLAUDE.md addition

```md
## Changelog discipline

When you ship a feature or fix, append an entry to `src/data/changelog.ts`:

- Bump the version: patch for fixes (`0.2.0` → `0.2.1`), minor for features (`0.2.0` → `0.3.0`), major for big shifts.
```

#### `cmpVersion`

Lives in `src/lib/version.ts` (new, tiny):

```ts
export function cmpVersion(a: string, b: string): number {
  const pa = a.split('.').map(Number);
  const pb = b.split('.').map(Number);
  for (let i = 0; i < 3; i++) {
    const da = pa[i] ?? 0;
    const db = pb[i] ?? 0;
    if (da !== db) return da < db ? -1 : 1;
  }
  return 0;
}
```

### B.5 Edge cases

- **Modal-on-modal collision** if user is in EntrySheet when the auto-popup tries to fire. Mitigation: trigger only on initial Dashboard mount, not on tab return. Use a `$state` flag scoped to the page lifetime so re-entering Dashboard doesn't re-fire.
- **Telegram BackButton conflict** with the existing onboarding BackButton handlers. Mitigation: scope the modal's BackButton handler with the show/hide bracket pattern, mirroring Onboarding.
- **Returning user with `stored > APP_VERSION`** (downgrade — should not happen): silently sync to current to avoid an infinite-popup loop.
- **No items in delta** (e.g., only `fix` items between stored and current): write current, skip the modal — don't celebrate fixes.
- **First-time user finishing onboarding** lands on Dashboard with no `last_shown_changelog_version`. Trigger writes current silently → no modal. ✓
- **User taps Profile icon before the auto-popup ever fires.** `markSeen()` writes current, dot clears, auto-popup will not fire on this session.

### B.6 Risks

- **Bundle size.** `ChangelogModal` could grow with backlog. Mitigation: dynamic import; tree-shake unused icons via the explicit icon whitelist.
- **Version drift between `package.json` and `changelog[0].version`.** Mitigation: CLAUDE.md instructs the LLM to keep them in sync; runtime never depends on `package.json` so a drift is cosmetic, not functional.
- **`<details>` styling cross-browser.** Mitigation: hide the default marker (`details > summary { list-style: none; }` + WebKit reset), render Lucide `ChevronDown` rotated via `[open]` attribute selector.
