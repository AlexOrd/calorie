# Calorie v0.3.0 — tracking dimensions, motivation, polish, sharing

**Date:** 2026-04-29
**Status:** spec
**Drivers:** user-requested feature push covering Telegram-native polish (without bot-side infra), motivation/gamification, new tracking dimensions, smarter logging shortcuts, and a shareable weekly summary.

---

## Overview

Ten features grouped into four waves. Each wave can ship and verify independently — the plan that follows this spec orders tasks so a wave-at-a-time subagent run is feasible without breaking master.

**In scope (10):**

| Wave                     | Feature                                                                                   |
| ------------------------ | ----------------------------------------------------------------------------------------- |
| 1 — Tracking foundations | C.1 Weight history · C.2 Body measurements · B.3 Projected goal date · D.2 Meal templates |
| 2 — Motivation           | B.1 Streaks · B.2 Weekly milestones · B.4 Daily summary                                   |
| 3 — Polish               | A.2 Add-to-Home-Screen · A.3 Biometric lock                                               |
| 4 — Sharing              | F.1 Save weekly card                                                                      |

**Explicitly out of scope:** A.1 MainButton (dropped — confusing with EntrySheet); fullscreen toggle; sleep/mood; voice input; smart-suggestion engine; photo journal; bot-DM reminders; `shareMessage` (needs bot infra); cross-device photo sync.

**Cross-platform requirement.** The app runs in Telegram (mobile + desktop) and as a PWA in browsers. Every feature must work in all three environments where the platform allows it. Two features carry platform branches (A.2 Add-to-Home-Screen and A.3 Biometric lock) — both isolated behind unified wrappers in `$lib/install` and `$lib/biometric` respectively.

---

## Goals

- Surface the existing data more powerfully: streaks, milestones, projected goal date, daily summary, weight chart all _derive_ from data the app already collects.
- Add two new tracking dimensions (weight history, body measurements) without expanding the core food-logging concept.
- Reduce daily friction with meal templates ("save current day, replay tomorrow").
- Treat the platform's polish APIs (Add-to-Home-Screen, biometric lock) as table stakes, but degrade silently where they aren't supported.
- Ship a shareable weekly summary card as a PNG download — no third-party image host, no bot.

## Non-goals

- A bot worker (no DMs, no shareMessage flow).
- Cross-device sync of large binary blobs (photos).
- Voice input or any ML-based food recognition.
- Custom themes / multi-language UI (still Ukrainian-only).
- Restructure of the core 8-category food model.

---

## Tooling decisions

- **`html-to-image`** (~30 KB, no transitive deps) — the only new runtime dependency. Used by `$lib/weeklyCard` to render a Svelte component to a PNG. Picked over `html2canvas` because it has 1-for-1 fidelity, zero dependencies, and produces cleaner output for Tailwind/Lucide content.
- **No new chart library.** Custom SVG continues for the new weight-history line chart, mirroring the existing `CategoryBarChart` pattern (~40 lines of SVG, theme-token aware). LayerChart was considered but rejected — too much bundle for a single line chart.
- **No new motion / UI lib.** Existing `motion` + `canvas-confetti` + Melt UI cover all animation, modal, and sheet needs.

## Decisions

### C.1 Weight history

**Storage.** New key `weight_log` shaped as `Record<YYYY-MM-DD, number>`. JSON-serialised in CloudStorage / LocalForage transparently. Fits ~150 daily entries per 4 KB Telegram CloudStorage key — enough for a year. Sharding to `weight_log_YYYY` is a future option but not needed in v0.3.0.

**Input.** Profile gets a "Додати вагу сьогодні" pill button next to the existing weight field. Clicking it sets today's entry from the current `profile.weight`. Editing the weight field separately _also_ writes to `weight_log[today]` so any save acts as a log.

**Chart.** A new `WeightHistoryChart.svelte` (custom SVG, last 90 days). Includes:

- Line through every logged day, dots on actual entries.
- Horizontal target overlay if `profile.target_weight_kg` is set (dashed line + label).
- Area fill below the line in the accent colour at low opacity.
- Min/max labels at the y-axis edges.

**Profile placement.** Below the existing macro targets card.

### C.2 Body measurements

**Inputs.** Optional fields on `UserProfile`: `waist_cm`, `neck_cm`, `hip_cm` (the last only required for `gender === 'female'`). Added to the existing `ProfileForm` as an expandable "Антропометрія" group so the form stays compact for users who don't care.

**Computation.** US Navy body-fat formula, implemented in `$lib/bodyfat.ts`:

```
Male:   BF% = 86.010 × log10(waist − neck) − 70.041 × log10(height_cm) + 36.76
Female: BF% = 163.205 × log10(waist + hip − neck) − 97.684 × log10(height_cm) − 78.387
```

Returns `null` when any required input is missing or yields a non-finite value.

**Display.** A small pill on the Stats `EnergyBalanceRow` (full variant) sitting next to the BMI pill, only when the formula returns a value: `Жир 17.4 %`.

### B.3 Projected goal date

**Input.** New optional `target_weight_kg` field on `UserProfile`. Added to `ProfileForm` next to the weight field (small, optional).

**Computation.** Pure function in `$lib/projection.ts`:

```
delta_kg = current_weight − target_weight        (sign matters: positive = need to lose)
avg_daily_kcal = mean(actualBurn − intake) over last 7 days where data exists
days_to_target = (delta_kg × 7700) / avg_daily_kcal
```

Returns `{ etaIso: string, daysAway: number }` if the user is on a trajectory toward the target (correct sign), otherwise `null`.

**Display.** A "Прогноз" card on the Profile page beneath the macro-targets card. Shows:

- The current trajectory and ETA: "За поточним темпом ≈ 75 кг до 15 черв. 2026 (47 днів)".
- A muted line below: "На основі середнього дефіциту 530 ккал/добу за 7 днів".
- If projection is `null`: "Встанови цільову вагу, щоб побачити прогноз" _or_ "Не вистачає даних для прогнозу — додай записи в журнал".

### D.2 Meal templates

**Storage.** New key `meal_templates` shaped as `MealTemplate[]`:

```ts
interface MealTemplate {
  id: string; // crypto.randomUUID()
  name: string; // user-supplied
  items: { cat: CategoryKey; id: string; pct: number }[];
}
```

**Save flow.** A new "Шаблони" button on the Journal route header opens a sheet with two tabs: "Мої шаблони" and "Зберегти поточний день". The save tab shows a preview of today's entries grouped by category, a name field, and a "Зберегти" button.

**Apply flow.** From the "Мої шаблони" tab, each template card shows name + a compact list of items. A primary "Додати в журнал" button creates one new `LogEntry` per template item, all with current timestamp. A muted "Видалити" button on each card with a confirm.

**No edit flow in v0.3.0** — templates are immutable after save (delete + re-save). Keeps scope tight.

### B.1 Streaks

**Computation.** Pure function in `$lib/streaks.ts`. Loads up to 365 days of log + activity records, walks back from today, and counts:

- **Deficit streak:** consecutive days where `intake − actualBurn < −NEUTRAL_BAND_KCAL`.
- **Water streak:** consecutive days where `waterMl >= hydrationTarget(profile)`.
- **Category streak:** consecutive days where `dayVerdict === 1` (no category over 100 %).

For each, returns `{ current: number, best: number }`. A day with no data breaks the streak (no grace).

**Display.** A new `StreaksRow.svelte` mounted at the top of Dashboard, above `EnergyBalanceRow`. Three pills horizontally:

- 🔥 Дефіцит · `7 днів` (best 12)
- 💧 Вода · `3 дні` (best 6)
- 🥗 Категорії · `0 днів` (best 4)

Pills use the same theme-token palette as elsewhere. Each pill is tappable to expand inline with a one-line explanation of the rule.

### B.2 Weekly milestones

**Trigger.** On Dashboard mount, when:

1. Today is Sunday (or Monday morning before any entry — pick Monday for resilience).
2. `lastCelebratedWeek` storage key is older than the just-completed ISO week.

**Criteria** (any one is enough to celebrate):

- ≥ 5 deficit days in the last 7.
- All 7 hydration targets hit.
- 0 category strikes across all 7 days.

**Modal.** New `WeeklyMilestoneModal.svelte` — confetti on open (`burstConfetti`), success haptic, hero card per achieved badge:

| Badge | Title              | Body                                  |
| ----- | ------------------ | ------------------------------------- |
| 🔥    | Тиждень дефіциту   | "5+ днів у дефіциті — впевнений курс" |
| 💧    | Гідрований тиждень | "Ціль по воді щодня"                  |
| 🥗    | Чисті категорії    | "Тиждень без перевищень"              |

**State.** New storage key `last_celebrated_week` storing an ISO-week id (`2026-W17`). Updated on modal close.

### B.4 Daily summary

**Computation.** Pure derived state from the existing dailyLog + activity stores. No new storage.

**Display.** A new `DailySummaryCard.svelte` at the _bottom_ of the Dashboard route, below the category grid. Shows:

- Today's energy delta (with state colour).
- Streak status — three small icons + counts in one line.
- Water % of target.
- Top consumed category (the one closest to / over 100 %).
- Single sentence below: "8 категорій · 1 запис · 0 ккал" sort of summary.

Compact card (~120 px) to keep the route uncluttered.

### A.2 Add-to-Home-Screen

**Wrapper.** New `$lib/install.ts` exposes:

```ts
type InstallState = 'unsupported' | 'unknown' | 'added' | 'missed';
export function installState(): Promise<InstallState>;
export function promptInstall(): Promise<'added' | 'dismissed' | 'unsupported'>;
```

Internally:

- **Telegram:** uses `Telegram.WebApp.checkHomeScreenStatus(callback)` and `addToHomeScreen()` when present.
- **PWA:** intercepts `beforeinstallprompt` event at boot in `main.ts`, stashes the deferred event, and exposes a manual `prompt()` call. Detects already-installed via `matchMedia('(display-mode: standalone)').matches` and `getInstalledRelatedApps()` where supported.

**Trigger.** On the _third_ Dashboard mount, if `installState() === 'missed' || 'unknown'` and the `home_screen_prompt_shown` storage key is `null`, show a small in-app banner offering installation. On tap → `promptInstall()`. On dismiss → write `home_screen_prompt_shown = { seenAt }` so it never fires again.

**Banner UI.** A light bg-accent/10 card above `StreaksRow` with a Lucide `Smartphone` icon, one-line title ("Встанови як додаток"), one-line description, primary button "Встановити", muted dismiss button.

### A.3 Biometric lock

**Wrapper.** New `$lib/biometric.ts`:

```ts
type BiometricSupport = 'telegram' | 'webauthn' | 'unsupported';
export function biometricSupport(): Promise<BiometricSupport>;
export function enrol(): Promise<boolean>; // returns true on success
export function verify(): Promise<boolean>;
```

- **Telegram:** uses `BiometricManager.init` → `requestAccess` → `authenticate` flow. On enrol, also stores a non-empty token via `updateBiometricToken('1')` so subsequent `authenticate` calls succeed.
- **WebAuthn:** generates a local 32-byte challenge, calls `navigator.credentials.create({ publicKey: { authenticatorSelection: { userVerification: 'required', authenticatorAttachment: 'platform' } } })`. Stores the resulting credential id in CloudStorage (under `biometric_credential_id`). On verify, calls `navigator.credentials.get` with the stored id. We don't verify the cryptographic signature — the OS biometric prompt is the only meaningful security boundary for a personal app without a server.
- **Unsupported:** all calls reject with `'unsupported'`. Profile toggle is hidden.

**Profile toggle.** A new "Захист Face ID / Touch ID" row on Profile, only visible when `biometricSupport()` is anything other than `'unsupported'`. Toggle on → calls `enrol()` and persists `profile.biometric_lock = true`. Toggle off → clears the credential id and the profile flag.

**Lock overlay.** New `BiometricLockOverlay.svelte` mounted in `App.svelte`. Activates when:

- App boots and `profile.biometric_lock === true`.
- App regains focus after 5 minutes of inactivity (`document.visibilitychange` + idle timer).

The overlay is a full-screen blur with a "Розблокувати" button + the appropriate icon. On mount it auto-calls `verify()`. On success, removes itself.

**Failure handling.** On verify failure, the overlay keeps the button visible — the user can retry. After 3 consecutive failures, an inline "Видалити блокування" link appears that disables the lock and clears the credential.

### F.1 Save weekly card

**Component.** New `WeeklyCardPreview.svelte` rendered off-screen at `position: fixed; left: -9999px; top: 0; width: 1080px; height: 1920px; pointer-events: none;`. Always rendered so we can snapshot it on demand.

**Layout** (described in the brainstorming notes):

```
┌──────────────────────────────────────────┐
│ Header: gradient + brand                 │
│ "Тиждень XX–XX"                          │
│                                          │
│ −2 340 ккал                              │
│ за 7 днів                                │
│                                          │
│ [7 mini bars: per-day delta]             │
│                                          │
│ 🔥 5 днів дефіциту                       │
│ 💧 6/7 водних цілей                      │
│ 🥗 4 чисті категорії                     │
│                                          │
│ Calorie · 0.3.0                          │
└──────────────────────────────────────────┘
```

Tailwind for styling (uses theme tokens — exports the _current_ theme), Lucide icons, system-ui fonts. Linear gradient background (`from-accent/10 via-bg to-warn/5`), large tabular delta number, day bars centred on a baseline.

**Export.** `$lib/weeklyCard.ts` exposes:

```ts
export async function exportWeeklyCard(node: HTMLElement): Promise<Blob>;
```

Uses `html-to-image`'s `toBlob(node, { width: 1080, height: 1920, pixelRatio: 1 })`. Stats page mounts the preview component (off-screen) and a button "Зберегти підсумок тижня". On tap: `exportWeeklyCard` → create object URL → trigger download via temporary `<a download>` element.

**Filename.** `calorie-week-2026-04-29.png` (current date suffix).

### Data flow

```
profile + activity + dailyLog stores (already exist)
    │
    ├─→ streaks.ts → StreaksRow + DailySummaryCard
    ├─→ projection.ts → ProjectionCard on Profile
    ├─→ bodyfat.ts → BMI/Body-fat pill on Stats
    └─→ weeklyCard.ts → off-screen Svelte preview → html-to-image → PNG download

weight_log key → weightLog store → WeightHistoryChart
meal_templates key → templates.ts → MealTemplatesSheet
```

### Storage keys (new)

| Key                        | Shape                                                           |
| -------------------------- | --------------------------------------------------------------- |
| `weight_log`               | `Record<YYYY-MM-DD, number>`                                    |
| `meal_templates`           | `MealTemplate[]`                                                |
| `home_screen_prompt_shown` | `{ seenAt: string } \| null`                                    |

### Profile migration

`profile.svelte.ts` `load()` reads existing `UserProfile`. New optional fields default to `undefined`. Saving any profile rewrites the whole record, so future loads always include the new shape. No version field needed.

## Edge cases

- **No profile yet (onboarding).** All new components gate on `profile.value` being non-null. Onboarding screen is unchanged.
- **Empty data history.** Streaks / projection / weight chart all render an empty state with a one-line hint to start logging.
- **Telegram Desktop without biometric hardware.** `BiometricManager.isBiometricAvailable === false`. Wrapper reports `'unsupported'`; toggle hidden.
- **PWA in Safari without WebAuthn platform authenticator.** `navigator.credentials` exists but `create()` throws. Wrapper reports `'unsupported'`; toggle hidden.
- **`html-to-image` failure** (e.g. tainted canvas due to a cross-origin image). The card uses zero remote images — only Lucide icons (inline SVGs) and theme colours — so this should not happen. If it does, surface a `alertAsync` with a clear message; do not crash.
- **CloudStorage 4 KB cap on `meal_templates`.** A user with 50 templates × ~50 bytes each ≈ 2.5 KB. Fine. Add a soft warning at 30 templates.
- **Weight log size.** ~150 entries per 4 KB, so ~5 months of daily logs. Acceptable for v0.3.0; shard to `weight_log_YYYY` only if a real user hits the cap.
- **Idle-detection across tabs.** Use `document.visibilitychange` and a 5-min idle timer scoped to the App.svelte component lifetime. The lock is best-effort; a user actively interacting with the page won't see the lock.

## Verification

- `command pnpm run check` — 0 errors / 0 warnings.
- `command pnpm run lint` — 0 errors / 0 warnings.
- `command pnpm run build` — succeeds. New runtime dep (`html-to-image`) accounted for in the bundle size.
- Manual viewport pass on iPhone 14 Pro Max / Galaxy S25 / iPhone SE for: weight chart, projection card, streaks row, milestone modal trigger, install banner, biometric lock overlay (Telegram + PWA), meal templates sheet, weekly card export.
- Telegram-mobile + Telegram-desktop + Chrome-PWA smoke test, paying attention to the platform-branched features (A.2, A.3).

## Risks

- **Biometric WebAuthn complexity.** The cryptographic verification is genuinely simplified (we trust the OS prompt, not the signature). For a personal app with no server this is the right tradeoff, but a security audit would push back. Document the simplification in `$lib/biometric.ts` JSDoc.
- **`html-to-image` bundle weight.** ~30 KB gzipped. Measured against the existing 274 KB main bundle, +11 % is acceptable. Lazy-import the lib only when the export button is tapped to keep the cold-boot bundle untouched.
- **Profile field sprawl.** v0.3.0 adds 4 optional fields. The form needs careful grouping ("Антропометрія") to stay readable.
- **Idle-lock UX.** Aggressive locking irritates users. 5 min is a deliberate compromise; surface the value as a constant in the wrapper so it can move easily.
